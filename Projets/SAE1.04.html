Création d'une base de données

Mission :

Création d'une base de données afin d'analyser une situation (naufrage du Titanic).

But :

Concevoir d'A à Z une base de données SQL et l'exploiter pour récupérer des informations.

Nombre de participants :

2

Travail effectué par tout le monde :

Il nous a fallu dans un premier temps définir des règles de gestion à partir du sujet qui définissent comment concevoir le reste de la base de données. Ces règles de gestion permettent de poser des contraintes telles que les relations entre les différentes tables de la base de données, la multiplicité ou le type d'un attribut ou encore quels attributs regrouper dans une même table.

Nous avons ensuite conçu et justifié un schéma entité association (SEA, représentation sur un schéma des différentes tables de la base de données avec leurs attributs et les relations entre ces tables) à partir des règles de gestion. Ce SEA avait pour rôle de produire ensuite un schéma relationnel (SLR, représentation des tables par une ligne de texte exprimant les attributs de chaque table ainsi que les attributs qui servent de clés primaires ou étrangères. Le SLR a pour vocation d'être implémenté dans la base de données tel quel) en appliquant des règles strictes. Le SLR nous a été fournit dans le cadre du projet et il nous a fallu l'implémenter sur PostgreSQL. Nous avons tout de même dû justifier les choix qui ont conduit au SLR fournit.

Finalement, nous avons écrit des requêtes en langage SQL afin d'exploiter la base de données et rendre un cours compte rendu de ce que nous avions conclu du naufrage grâce à ces données.

Travail que j'ai effectué :

J'ai tout d'abord conçu le SEA à partir des informations données par le sujet du projet. Il m'a fallu pour cela interpréter les indications données dans le sujet et m'aider des données fournies, qui allaient ensuite être intégrées dans notre base de données. Il m'a surtout fallu déterminer quels noms donner aux tables et aux attributs afin qu'ils soient compréhensibles, de même que les associations entre les tables et leurs cardinalités. Bien définir cela est primordial afin que le schéma soit bien compréhensible et utilisable pour mettre en place la base de données. Il faut être capable de manipuler les données comme souhaité.

J'ai ensuite justifié le schéma relationnel proposé dans le cadre du projet : type de données, tables choisies, contraintes, etc. comme nous aurions du le faire si nous l'avions conçu nous-même. Pour cela, je me suis appuyé sur des régles qui définissent comment traduire un SEA en SLR, en j'ai interprété les informations quand nécessaire.

Enfin, j'ai écrit une partie des requêtes en langage SQL afin d'exploiter les informations rentrées dans la base de donnée pour répondre au sujet. Pour terminer, j'ai rédigé le compte rendu de l'exploitation de la base de données afin de répondre au sujet du projet.

Compétences du programme national :

- Mettre à jour et interroger une base de données relationnelle (en requêtes directes ou à travers une application) - Visualiser des données - Concevoir une base de données relationnelle à partir d’un cahier des charges
